1. PROJECT OVERVIEW
================================================================================

What the project is:
This is a full-stack web application called "Store Task & Checklist Manager"
built with Spring Boot (Java) on the backend and React (JavaScript) on the
frontend. It is an internal workflow management system designed for retail
store operations.

Main goal:
To help store teams manage their daily tasks, checklists, shift handovers,
projects, and team assignments in one centralized platform. It replaces
paper-based or spreadsheet-based tracking with a digital system.

Main features:
- User authentication with JWT tokens
- Role-based access control (6 roles)
- Task management (CRUD, status tracking, priority)
- Project management
- Team management with member assignment
- Shift management
- Checklist creation with trackable items
- Shift handover management with resolve/acknowledge flow
- Dashboard with statistics and KPIs
- Note taking on tasks, checklist items, and handovers
- Audit logging for all actions
- Password management (change/reset)

Problem it solves:
Retail stores often have multiple shifts, teams, and a large number of daily
operational tasks. Without a centralized system, tasks get lost, handovers
are missed, and there is no accountability. This system ensures every task
is tracked, every handover is acknowledged, and every shift knows what needs
to be done.

Intended users:
- SUPER_ADMIN: Full system access, can manage everything
- MANAGER: Can manage projects, teams, shifts, users, tasks
- TEAM_LEAD: Can manage tasks within their team, view team members
- DEVELOPER: Can create checklists, view tasks, participate in workflow
- TESTER: Can create checklists, view tasks, participate in workflow
- STAFF: Can create checklists, complete checklist items, view tasks

Overall workflow:
1. Admin creates users with specific roles and assigns them to teams/shifts.
2. Managers create projects and teams, assign team leads and members.
3. Team leads/managers create tasks and assign them to team members.
4. Any user can create checklists with items for their shift/team.
5. Users complete checklist items and mark tasks as they progress.
6. During shift changes, users create handovers documenting completed work,
   pending work, and blockers for the next shift.
7. The receiving shift can resolve and acknowledge handovers.
8. The dashboard provides a real-time overview of all activities.

2. TECH STACK
================================================================================

FRONTEND:
---------
React (v18.2.0):
  - A modern JavaScript library for building user interfaces.
  - Chosen for its component-based architecture, reusability, and vast ecosystem.

Vite (v5.0.8):
  - A fast build tool and development server.
  - Chosen over Create React App because it provides instant server start,
    fast hot module replacement (HMR), and optimized production builds.

Tailwind CSS (v3.4.0):
  - A utility-first CSS framework.
  - Chosen because it allows rapid UI development without writing custom CSS,
    provides consistent design, and keeps the bundle small.

Axios (v1.6.2):
  - A promise-based HTTP client for the browser.
  - Chosen for making API requests with ease. Supports interceptors for
    automatic JWT token attachment and error handling.

React Router DOM (v6.21.0):
  - Declarative routing for React applications.
  - Chosen for handling navigation between pages, protected routes, and
    nested layouts.

BACKEND:
--------
Spring Boot (v3.2.0):
  - A Java framework for building production-ready web applications.
  - Chosen for its auto-configuration, embedded server, and enterprise-grade
    capabilities. Provides REST API support out of the box.

Spring Security:
  - A powerful authentication and authorization framework.
  - Chosen to implement JWT-based authentication, role-based access control,
    and secure API endpoints.

JWT (JSON Web Tokens) via jjwt library:
  - A compact, URL-safe token format for authentication.
  - Chosen because it allows stateless authentication - no session storage
    needed. Tokens contain all user information needed for authorization.

JPA / Hibernate:
  - Java Persistence API with Hibernate as the ORM provider.
  - Chosen for object-relational mapping - it lets us work with database
    tables as Java objects, auto-generates SQL, and manages relationships.

Spring Boot Starter Validation:
  - Provides Bean Validation API (Jakarta Validation).
  - Chosen to validate incoming request data with annotations like @NotBlank,
    @Email, etc.

Lombok:
  - A Java library that reduces boilerplate code.
  - Chosen to auto-generate getters, setters, constructors via annotations
    like @Data, @RequiredArgsConstructor.

DATABASE:
--------
PostgreSQL:
  - A powerful, open-source relational database.
  - Chosen for its reliability, support for complex queries, ACID compliance,
    and excellent JSON support. Well-suited for enterprise applications.

--- FILE: src/main.jsx ---
Purpose: Entry point of the React application. Renders the App component
into the DOM element with id "root".
What it does: Calls ReactDOM.createRoot to mount the App component inside
React.StrictMode (which helps catch bugs in development).

--- FILE: src/App.jsx ---
Purpose: Root component that sets up routing and authentication context.
Main Logic: Wraps everything in AuthProvider (for auth state), then
BrowserRouter. Defines routes:
  - /login  → Login page (public)
  - /       → ProtectedRoute wrapping Layout (authenticated users only)
    - /dashboard  → Dashboard page
    - /users      → Users page
    - /teams      → Teams page
    - /projects   → Projects page
    - /tasks      → Tasks page
    - /shifts     → Shifts page
    - /checklists → Checklists page
    - /handovers  → Handovers page
  - Default redirect from / to /dashboard
Routing: Uses react-router-dom v6 with nested routes. Layout component
contains sidebar and renders child pages via <Outlet />.
Role-based rendering: Not at this level - individual pages handle their own
role checks.
Connected Components: Layout, ProtectedRoute, Login, Dashboard, Users,
Teams, Projects, Tasks, Shifts, Checklists, Handovers.

--- FILE: src/index.css ---
Purpose: Global CSS styles.
What it defines:
  - Tailwind directives (@tailwind base, components, utilities)
  - Base reset (margin, padding, box-sizing)
  - Body styles (font-family, background color)
  - Utility classes: .sidebar-link, .btn, .btn-primary, .btn-secondary,
    .btn-danger, .input, .card, .table, .badge with color variants
    (.badge-blue, .badge-green, .badge-yellow, .badge-red, .badge-gray,
     .badge-purple, .badge-orange)
Connected to: All pages use these CSS classes for consistent styling.

--- FILE: src/services/api.js ---
Purpose: Central API service module. Creates an Axios instance configured
to communicate with the backend at '/api' (proxied via Vite).
Main Logic:
  - Creates axios instance with baseURL '/api' and JSON content type
  - Request interceptor: Attaches JWT token from localStorage as
    'Authorization: Bearer <token>' header to every request
  - Response interceptor: On 401 error, clears localStorage and redirects
    to /login page
Exports service objects for each API domain:
  - authService: login, changePassword, createUser, getCurrentUser
  - userService: getAll, getById, getByTeam, getByRole, update, resetPassword,
    getProfile, updateProfile, changePassword
  - teamService: getAll, getById, getByProject, getMyTeam, create, update, delete
  - projectService: getAll, getById, getByStatus, create, update, delete
  - taskService: getAll, getById, getByProject, getByTeam, getByUser,
    getMyTasks, getMyPending, getOverdue, create, update, delete
  - shiftService: getAll, getById, getActive, create, update, delete
  - checklistService: getAll, getById, getByShift, getByTeam, create,
    completeItem, uncompleteItem, delete
  - handoverService: getAll, getById, getByShift, getUnresolved, getByTeam,
    getUnacknowledged, create, resolve, acknowledge, delete
  - noteService: getByTask, getByChecklistItem, getByHandover, create, delete
  - dashboardService: getData
  - auditService: getAll, getRecent, getByUser
Connected Backend API: All endpoints under /api/*. Each service function maps
to a backend REST endpoint.

--- FILE: src/context/AuthContext.jsx ---
Purpose: Provides authentication state management across the entire app.
State handling:
  - user: Current user object (null when not logged in)
  - loading: Boolean indicating if auth state is being restored from localStorage
Main functions:
  - login(username, password): Calls authService.login, stores token and user
    data in localStorage, sets user state.
  - logout(): Clears localStorage, sets user to null.
  - changePassword(data): Calls authService.changePassword, updates
    user.firstLogin to false in localStorage.
  - isAuthenticated: Derived boolean (!!user)
  - On mount: Restores token and user from localStorage if they exist.
Connected Components: Used by all components via useAuth() hook.

--- FILE: src/components/ProtectedRoute.jsx ---
Purpose: Guards routes that require authentication.
Main Logic:
  - If loading → shows "Loading..." screen
  - If not authenticated → redirects to /login
  - If authenticated → renders child components
Connected Components: Wraps Layout in App.jsx.

  - initializeProjects(): Creates 3 projects

Relationships: References all entities and repositories. Assigns users to

Purpose: Configures Spring Security for the application.
Main beans:

  - DaoAuthenticationProvider: Links UserDetailsService with PasswordEncoder
  - AuthenticationManager: Handles authentication requests
  - CorsConfigurationSource: Allows requests from localhost:5173 and :3000
  - SecurityFilterChain: Configures HTTP security:
    * Disables CSRF (stateless API)
    * Sets session management to stateless
    * Permits /api/auth/** without authentication
    * Permits OPTIONS requests (CORS preflight)
    * All other requests require authentication
    * Adds JwtAuthenticationFilter before UsernamePasswordAuthenticationFilter
Security handling: Stateless JWT-based authentication. No sessions.

--- FILE: security/JwtTokenProvider.java ---
Purpose: Generates and validates JWT tokens.
Main methods:
  - init(): Creates HMAC-SHA key from the secret (pads to 32 bytes if needed)
  - generateToken(): Creates JWT with username as subject, issue/expiry dates
  - getUsernameFromToken(): Extracts username from JWT claims
  - validateToken(): Parses and verifies JWT signature and expiration
Configuration: JWT secret and expiration (24 hours) from application.properties

--- FILE: security/JwtAuthenticationFilter.java ---
Purpose: Intercepts every HTTP request to validate JWT token.
Main method:
  - doFilterInternal(): Extracts Bearer token from Authorization header,
    validates it, loads user details, and sets authentication in
    SecurityContextHolder.
  - getJwtFromRequest(): Parses "Bearer <token>" format from header.
Flow: Request → JwtAuthenticationFilter → validate token → set auth context →
request proceeds to controller.

6. DATABASE DESIGN

All tables are auto-created by Hibernate with ddl-auto=update.

TABLE: users
Columns:
  id            BIGINT (PK, auto-increment)
  username      VARCHAR (unique, not null)
  password      VARCHAR (not null)
  first_name    VARCHAR (not null)
  last_name     VARCHAR (not null)
  email         VARCHAR (unique, not null)
  phone         VARCHAR
  status        VARCHAR (default 'ACTIVE')
  role          VARCHAR (enum as string, not null)
  first_login   BOOLEAN (default true)
  team_id       BIGINT (FK → teams.id, nullable)
  shift_id      BIGINT (FK → shifts.id, nullable)
  created_at    TIMESTAMP
  updated_at    TIMESTAMP
Relationships:
  - Many users → One team (team_id FK)
  - Many users → One shift (shift_id FK)

TABLE: teams
Columns:
  id              BIGINT (PK, auto-increment)
  name            VARCHAR (not null)
  description     TEXT
  project_id      BIGINT (FK → projects.id, nullable)
  team_lead_id    BIGINT (FK → users.id, nullable)
  manager_id      BIGINT (FK → users.id, nullable)
  created_at      TIMESTAMP
  updated_at      TIMESTAMP
Relationships:
  - Many teams → One project (project_id FK)
  - Many teams → One user as team lead (team_lead_id FK)
  - Many teams → One user as manager (manager_id FK)
  - One team → Many users (members)
  - One team → Many tasks
  - One team → Many shifts

TABLE: projects
Columns:
  id            BIGINT (PK, auto-increment)
  name          VARCHAR (not null)
  description   TEXT
  status        VARCHAR
  start_date    DATE
  end_date      DATE
  created_at    TIMESTAMP
  updated_at    TIMESTAMP
Relationships:
  - One project → Many teams
  - One project → Many tasks

TABLE: tasks
Columns:
  id              BIGINT (PK, auto-increment)
  title           VARCHAR (not null)
  description     TEXT
  status          VARCHAR (enum as string, not null, default 'TODO')
  priority        VARCHAR (enum as string, not null, default 'MEDIUM')
  project_id      BIGINT (FK → projects.id, nullable)
  team_id         BIGINT (FK → teams.id, nullable)
  assigned_to_id  BIGINT (FK → users.id, nullable)
  created_by_id   BIGINT (FK → users.id, nullable)
  due_date        DATE
  created_at      TIMESTAMP
  updated_at      TIMESTAMP
Relationships:
  - Many tasks → One project
  - Many tasks → One team
  - Many tasks → One user (assignedTo)
  - Many tasks → One user (createdBy)
  - One task → Many notes

TABLE: shifts
Columns:
  id            BIGINT (PK, auto-increment)
  name          VARCHAR (not null)
  shift_type    VARCHAR (enum as string, not null)
  start_time    TIME
  end_time      TIME
  active        BOOLEAN (default true)
  team_id       BIGINT (FK → teams.id, nullable)
  created_at    TIMESTAMP
  updated_at    TIMESTAMP
Relationships:
  - Many shifts → One team
  - One shift → Many users
  - One shift → Many checklists (as shift's checklist)
  - One shift → Many handovers (as fromShift or toShift)

TABLE: checklists
Columns:
  id              BIGINT (PK, auto-increment)
  title           VARCHAR (not null)
  description     TEXT
  shift_id        BIGINT (FK → shifts.id, nullable)

  - Many checklists → One shift
  - Many checklists → One team
  - Many checklists → One user (createdBy)
  - One checklist → Many checklist items

Columns:
  id                BIGINT (PK, auto-increment)

  completed         BOOLEAN (default false)

Columns:

================================================================================
7. COMPLETE WORKFLOW
================================================================================

A. LOGIN FLOW
1. User navigates to /login page.
2. User enters username and password, submits form.
3. Frontend calls POST /api/auth/login with { username, password }.
4. Backend AuthController receives request.
5. AuthService.login() is called:
   a. Looks up user by username from UserRepository.
   b. Calls AuthenticationManager.authenticate() which uses
      CustomUserDetailsService to load user and BCrypt to verify password.
   c. On success, JwtTokenProvider.generateToken() creates a JWT.
   d. Returns LoginResponse with token and user details.
6. Frontend AuthContext.login() stores token and user in localStorage.
7. User is navigated to /dashboard.
8. Subsequent requests automatically attach the JWT via Axios interceptor.

B. JWT AUTHENTICATION FLOW
1. Every API request goes through JwtAuthenticationFilter (OncePerRequestFilter).
2. Filter extracts "Bearer <token>" from Authorization header.
3. JwtTokenProvider.validateToken() verifies:
   a. Token signature using HMAC-SHA key.
   b. Token expiration date.
4. If valid, JwtTokenProvider.getUsernameFromToken() extracts username.
5. CustomUserDetailsService.loadUserByUsername() loads full user details.
6. UsernamePasswordAuthenticationToken is created with user and authorities.
7. SecurityContextHolder is set with this authentication.
8. The filter chain continues - controller can now use @AuthenticationPrincipal
   to access the authenticated user.

C. USER CREATION FLOW
1. Manager opens Users page, clicks "Add User".
2. Fills form: username, password, name, email, role, team.
3. Frontend calls POST /api/auth/create-user with CreateUserRequest body.
4. AuthController.createUser() is called with @PreAuthorize requiring
   SUPER_ADMIN or MANAGER role.
5. AuthService.createUser() does:
   a. Checks if username or email already exists (throws if so).
   b. Creates new User entity with BCrypt-encoded password.
   c. Sets firstLogin=true (forces password change on first login).
   d. Saves user via UserRepository.
   e. Logs audit entry "USER_CREATED".
6. Frontend refreshes user list.

D. TEAM ASSIGNMENT FLOW
1. Manager goes to Teams page, clicks "Add Team".
2. Selects project, manager (from MANAGER users), team lead (from TEAM_LEAD users),
   and checks members (STAFF/DEVELOPER/TESTER users).
3. Validates: manager != team lead, neither is in member list.
4. Frontend calls POST /api/teams with CreateTeamRequest.
5. TeamController.createTeam() (MANAGER+ role required).
6. TeamService.createTeam():
   a. Creates Team entity with project/manager/lead references.
   b. Saves team.
   c. For each memberId, finds user, sets user.team = team, saves user.
   d. Logs audit.
7. Users now belong to the team and can see team-related tasks/projects.

E. TASK CREATION FLOW
1. Manager or Team Lead opens Tasks page, clicks "Add Task".
2. Fills: title, description, status, priority, project, team,
   assigned user, due date.
3. Frontend calls POST /api/tasks with CreateTaskRequest.
4. TaskController.createTask() (MANAGER+ or TEAM_LEAD role).
5. TaskService.createTask():
   a. Creates Task entity.
   b. Resolves project/team/assignee FKs from IDs.
   c. Sets currentUser as createdBy.
   d. Saves task.
   e. Logs audit.
6. Assigned user can now see the task and update its status.

F. CHECKLIST WORKFLOW
1. Any user can create a checklist (all roles allowed).
2. Goes to Checklists page, clicks "Create Checklist".
3. Enters title, selects shift and team.
4. Adds items with titles and optional assignees.
5. Frontend calls POST /api/checklists with items array.
6. ChecklistService.createChecklist():
   a. Creates Checklist entity with cascade persistence of items.
   b. Each ChecklistItem is created with assigned user (if specified).
   c. Saves checklist (cascade saves all items).
7. Assignees see items on their shift's checklist.
8. User clicks checkbox to complete item:
   a. Frontend calls PUT /api/checklists/items/{id}/complete.
   b. ChecklistService.completeItem() sets completed=true, completedBy,
      completedAt.
9. Progress bar on the checklist card updates automatically.

G. HANDOVER WORKFLOW
1. At end of shift, user creates a handover.
2. Goes to Handovers page, clicks "Create Handover".
3. Fills: title, from/to shifts, teams, completed/pending work, blockers,
   instructions for next shift.
4. Frontend calls POST /api/handovers.
5. HandoverService.createHandover() saves with all relations.
6. Next shift sees the pending handover on their dashboard.
7. The oncoming shift worker clicks "Resolve" to acknowledge the content.
8. Then clicks "Acknowledge" to formally accept the handover.
9. Resolve: PUT /api/handovers/{id}/resolve → sets resolved=true.
10. Acknowledge: PUT /api/handovers/{id}/acknowledge → sets acknowledged=true
    + acknowledgedAt timestamp.

H. ROLE-BASED ACCESS WORKFLOW
1. When a user logs in, their role is returned as part of the JWT and user data.
2. Frontend uses role to conditionally show/hide UI elements:
   - Layout sidebar hides Users for non-managers/leads, hides Shifts for leads.
   - Pages check roles before showing Add/Edit/Delete buttons.
3. Backend enforces role checks via @PreAuthorize annotations:
   - @PreAuthorize("hasAnyRole('SUPER_ADMIN', 'MANAGER')") on sensitive endpoints.
   - @PreAuthorize("hasAnyRole('SUPER_ADMIN')") on delete endpoints.
4. If a user tries to access a restricted endpoint, Spring Security returns 403
   Forbidden, handled by GlobalExceptionHandler.

I. DASHBOARD DATA FLOW
1. User navigates to dashboard.
2. Frontend calls GET /api/dashboard.
3. DashboardController.getDashboard() gets current user.
4. DashboardService.getDashboardData(user):
   a. Determines if user is manager (full data) or staff (limited data).
   b. Managers: getAllTasks(), computes stats.
   c. Staff: Gets team tasks + assigned tasks (deduplicated), computes stats.
   d. Gets recent tasks (sorted by update time, top 10).
   e. If user has shift, gets pending handovers and pending checklist items.
   f. For leads+, gets missed handovers and pending approvals count.
5. Returns DashboardDTO with all computed stats.
6. Frontend renders stat cards and lists.

8. ROLE-BASED ACCESS CONTROL
================================================================================

SUPER_ADMIN:
  - Full system access
  - Can create/edit/delete everything: users, teams, projects, tasks,
    shifts, checklists, handovers
  - Can delete teams, projects, shifts (restricted to SUPER_ADMIN only)
  - Can view audit logs
  - Can reset any user's password
  - Sees all data on dashboard

MANAGER:
  - Can create/edit users (except SUPER_ADMIN)
  - Can create/edit/delete teams (but not delete - that's SUPER_ADMIN only)
  - Can create/edit projects
  - Can create/edit/delete tasks
  - Can create/edit shifts (but not delete - SUPER_ADMIN only)
  - Can delete checklists
  - Can view all users
  - Can view audit logs
  - Can reset user passwords
  - Sees all projects, teams, and tasks
  - Cannot delete teams/projects/shifts

TEAM_LEAD:
  - Cannot see/manage Users page (hidden from sidebar by Layout)
  - Cannot see Shifts page (hidden from sidebar)
  - Can create/edit/delete tasks within their team
  - Can view their team members
  - Can create checklists
  - Views only their team's data on dashboard
  - Cannot create/edit users

DEVELOPER:
  - Can create checklists and items
  - Can view tasks assigned to them or their team
  - Can complete checklist items
  - Can create handovers
  - Cannot create/edit users, teams, projects, shifts
  - Limited view on dashboard

TESTER:
  - Same as DEVELOPER - can create checklists, view tasks, complete items
  - Cannot manage users/teams/projects

STAFF:
  - Can create checklists and items
  - Can view assigned tasks
  - Can complete checklist items
  - Can create handovers
  - Cannot manage any entities
  - Most restricted view on dashboard

BACKEND RESTRICTION SUMMARY:
  - DELETE /api/teams/{id} → SUPER_ADMIN only
  - DELETE /api/projects/{id} → SUPER_ADMIN only
  - DELETE /api/shifts/{id} → SUPER_ADMIN only
  - POST/PUT /api/users/{id} → MANAGER+ only (edit user)
  - POST /api/users/{id}/reset-password → MANAGER+ only
  - POST /api/auth/create-user → MANAGER+ only
  - POST/PUT /api/projects → MANAGER+ only
  - POST/PUT /api/teams → MANAGER+ only
  - POST/PUT /api/shifts → MANAGER+ only
  - POST/DELETE /api/tasks → MANAGER+ or TEAM_LEAD
  - DELETE /api/checklists/{id} → MANAGER+ only
  - GET /api/audit-logs/** → MANAGER+ only
  - PUT /api/tasks/{id} → any authenticated user
  - POST /api/checklists → all authenticated users
  - POST /api/handovers → all authenticated users

11. AUTHENTICATION & SECURITY
================================================================================

JWT GENERATION:
  - Triggered by successful authentication via POST /api/auth/login.
  - AuthenticationManager authenticates username + password.
  - JwtTokenProvider.generateToken(authentication):
    * Extracts username from UserDetails principal.
    * Creates JWT with: subject=username, issuedAt=now, expiration=now+24h.
    * Signs with HMAC-SHA key derived from app.jwt.secret property.
    * Returns compact JWT string.

JWT VALIDATION:
  - JwtAuthenticationFilter intercepts every request (except /api/auth/**).
  - Extracts token from "Authorization: Bearer <token>" header.
  - Calls JwtTokenProvider.validateToken(token):
    * Parses JWT with the same signing key.
    * Validates signature (tamper-proof).
    * Validates expiration (rejects expired tokens).
  - If valid: extracts username, loads UserDetails, sets auth in context.
  - If invalid: continues filter chain without authentication (results in 401).

SPRING SECURITY FLOW:
  1. Request arrives at filter chain.
  2. JwtAuthenticationFilter.doFilterInternal() runs first.
  3. If JWT is present and valid:
     - Creates UsernamePasswordAuthenticationToken with
       userDetails.getAuthorities() which includes "ROLE_SUPER_ADMIN" etc.
     - Sets SecurityContextHolder authentication.
  4. Request reaches controller.
  5. @PreAuthorize annotations check authorities before method execution.
  6. If not authorized, AccessDeniedException is thrown → 403 response.

PASSWORD ENCODING:
  - BCryptPasswordEncoder is used (config in SecurityConfig).
  - Passwords are never stored in plain text.
  - During login: AuthenticationManager uses DaoAuthenticationProvider which
    calls PasswordEncoder.matches() to compare plaintext input with stored hash.
  - During user creation: password is encoded before saving.
  - During password change/reset: new password is encoded before saving.

PROTECTED ROUTES:
  - Authentication: Requests matching /api/auth/** are publicly accessible.
    All other routes require a valid JWT.
  - Authorization: @PreAuthorize annotations on controller methods:
    * hasAnyRole('SUPER_ADMIN', 'MANAGER') - manager-level access
    * hasAnyRole('SUPER_ADMIN', 'MANAGER', 'TEAM_LEAD') - lead-level access
    * hasAnyRole('SUPER_ADMIN', 'MANAGER', 'TEAM_LEAD', 'STAFF', 'DEVELOPER', 'TESTER')
      - all authenticated roles

ROLE AUTHORIZATION:
  - Spring Security grant authorities as "ROLE_SUPER_ADMIN", "ROLE_MANAGER", etc.
  - @PreAuthorize("hasAnyRole('SUPER_ADMIN', 'MANAGER')") internally checks for
    "ROLE_SUPER_ADMIN" or "ROLE_MANAGER" in the granted authorities.
  - Built via CustomUserDetailsService:
    Collections.singletonList(new SimpleGrantedAuthority("ROLE_" + user.getRole().name()))

12. IMPORTANT LEARNING NOTES
================================================================================

Why certain architecture is used:
  - Three-tier architecture (Controller → Service → Repository):
    * Separation of concerns - each layer has a specific responsibility.
    * Controllers handle HTTP requests/responses.
    * Services contain business logic.
    * Repositories handle database operations.
    * Makes the code testable, maintainable, and modular.

Why DTOs are used:
  - Data Transfer Objects separate the internal entity structure from the
    API response structure.
  - Entities may contain sensitive data (passwords) or circular references
    that should not be exposed.
  - DTOs allow combining data from multiple entities (e.g., TaskDTO includes
    projectName from Project entity).
  - DTOs prevent over-fetching or under-fetching of data.
  - DTOs act as a contract between frontend and backend.

Why Repositories are used:
  - Spring Data JPA repositories provide CRUD operations without boilerplate.
  - Custom query methods are derived from method names (e.g., findByTeamId).
  - @Query annotations allow custom JPQL/HQL queries.
  - Repositories handle transaction management and entity lifecycle.

Why JWT is used:
  - Stateless authentication - no server-side session storage needed.
  - Tokens contain all user information, reducing database lookups.
  - Easy to use in distributed/microservice architectures.
  - Frontend stores token in localStorage and sends with every request.
  - Works well with REST APIs (no cookies/sessions needed).

Why React state is needed:
  - Components need to respond to user interactions and data changes.
  - State allows tracking form inputs, API responses, UI visibility.
  - Context (AuthContext) provides global state without prop drilling.
  - useState handles local component state (form data, modals, loading).
  - useEffect handles side effects (API calls on mount).
  - Proper state management ensures UI reflects the current data.

Why APIs are separated:
  - Clean separation between frontend and backend.
  - Each API endpoint handles a specific resource/action.
  - RESTful naming convention (/api/users, /api/tasks, etc.).
  - Allows independent development and testing.
  - If needed, the backend can be replaced or multiple frontends can use it.

Why @Transactional is used:
  - Ensures database operations are atomic - all succeed or all roll back.
  - Critical for operations that modify multiple entities (e.g., creating a
    team and assigning members).
  - Prevents partial updates that could leave data in inconsistent state.

Why cascade = CascadeType.ALL is used on Checklist.items:
  - When saving a Checklist with new items, cascade ensures items are saved
    automatically.
  - When deleting a Checklist, cascade ensures all items are deleted.
  - Simplifies code by handling related entity lifecycle.

13. COMMON ISSUES & DEBUGGING
================================================================================

Issue: Cannot log in - "Invalid username or password"
  Cause: Wrong credentials, user not found, or password not matching.
  Fix: Verify username exists in users table. Verify password is BCrypt encoded.
       Check user status is 'ACTIVE' (not 'INACTIVE', 'SUSPENDED', 'LOCKED').

Issue: Account disabled/locked error
  Cause: User.status is not 'ACTIVE'.
  Fix: Update user status to 'ACTIVE' in database or via API.

Issue: JWT token expired
  Cause: Token validity is 24 hours (86400000ms).
  Fix: User needs to log in again. Frontend automatically redirects to /login
       on 401 response via Axios interceptor.

Issue: 403 Forbidden on API calls
  Cause: User role does not have permission for the endpoint.
  Fix: Check @PreAuthorize annotations on the controller method. Verify
       user's role matches the required role.

Issue: Duplicate username/email error (409 Conflict)
  Cause: Username or email already exists in the database.
  Fix: Use unique values. GlobalExceptionHandler detects the constraint
       violation and returns a user-friendly message.

Issue: Data integrity violation when deleting
  Cause: Trying to delete an entity that is referenced by other entities
         (e.g., deleting a project that has teams).
  Fix: Delete child entities first, or set FKs to null before deletion.

Issue: Invalid enum value error (400 Bad Request)
  Cause: Sending a value that doesn't match the enum (e.g., "URGENT" for
         TaskPriority which only allows LOW/MEDIUM/HIGH).
  Fix: GlobalExceptionHandler catches InvalidFormatException and shows
       allowed values in the error message.

Issue: End date before start date
  Cause: Validation in ProjectService and ShiftService catches this.
  Fix: Provide end date that is after start date. Frontend also validates.

Issue: Same person as manager and team lead
  Cause: TeamService.validateTeamAssignment() prevents this.
  Fix: Choose different users for manager and team lead roles.

Issue: Checklist progress not updating
  Cause: Progress is calculated from entity data, not cached. It's computed
         in the mapToDTO method.
  Fix: Ensure items are properly loaded (not lazy-loaded when DTO is mapped).
       The service uses @Transactional(readOnly = true) to ensure lazy loading works.

Issue: CORS errors in browser
  Cause: Frontend running on port 5173, backend on 8080.
  Fix: SecurityConfig configures CORS to allow localhost:5173 and :3000.
       Vite also proxies /api requests to the backend.

Issue: Frontend shows blank page after login
  Cause: Token not stored properly or user data not parsed from response.
  Fix: Check localStorage for 'token' and 'user' keys. Ensure the login
       response matches what AuthContext expects.

Issue: Axios interceptor not redirecting on 401
  Cause: The error response might not have status 401 (e.g., 403 or 500).
  Fix: The interceptor only checks for status === 401. Other errors should
       be handled by individual page error handlers.

Issue: First login password change required
  Cause: User.firstLogin is true, but the frontend doesn't enforce the change.
  Fix: The Layout component shows "Change Password" button, but doesn't force
       it. The backend doesn't block non-password-changed users.

Issue: Database tables not created
  Cause: spring.jpa.hibernate.ddl-auto is set to "update", not "create".
  Fix: Ensure PostgreSQL is running and the database "taskmanager" exists.
       Tables will be auto-created on first run.

14. FINAL PROJECT SUMMARY
================================================================================

Overall architecture:
  The project follows a modern full-stack architecture:
  - Frontend: Single Page Application (SPA) built with React, Vite, and
    Tailwind CSS. Communicates with backend via REST APIs.
  - Backend: Spring Boot REST API with three layers (Controller-Service-
    Repository). Uses JPA/Hibernate for database access and Spring Security
    for JWT-based authentication.
  - Database: PostgreSQL relational database with 10 tables connected through
    foreign key relationships.
  - Communication: Frontend ↔ Backend via REST JSON APIs with JWT auth.
    Backend ↔ Database via Hibernate ORM.

Strengths of project:
  - Comprehensive role-based access control with 6 distinct roles.
  - Complete CRUD for all major entities (users, teams, projects, tasks,
    shifts, checklists, handovers).
  - Checklist with progress tracking and item-level completion.
  - Handover workflow with resolve/acknowledge lifecycle.
  - Dashboard provides real-time KPIs tailored to each user's role.
  - Audit logging for all important actions.
  - JWT-based stateless authentication.
  - Seed data initializer for quick development setup.
  - Clean separation of concerns with layered architecture.
  - Validation at both frontend and backend levels.

Current limitations:
  - No email notification system for task assignments or handovers.
  - No file upload/attachment support.
  - No real-time updates (WebSocket/polling for live changes).
  - No pagination for large lists (all data loaded at once).
  - No search/filter functionality on list pages.
  - No automated testing (unit/integration tests are empty).
  - First login password change is not enforced (just a button).
  - No password strength requirements beyond minimum 6 characters.
  - No data export functionality (CSV/PDF).
  - No mobile-responsive design.
  - Note service is implemented but no frontend UI for creating/viewing notes.
  - Audit log viewer exists in backend but has no frontend UI.
  - No soft delete (entities are permanently deleted).
  - No caching mechanism for frequently accessed data.

Possible future improvements:
  - Add email notifications for task assignments and handover creation.
  - Implement file/image uploads for tasks and handovers.
  - Add WebSocket for real-time updates (when someone completes a checklist item).
  - Implement pagination and search/filter for all list pages.
  - Write comprehensive unit and integration tests.
  - Enforce password change on first login.
  - Add password strength requirements and validation.
  - Add data export functionality (report generation).
  - Make the UI fully responsive for mobile devices.
  - Build frontend UI for notes and audit log viewing.
  - Implement soft delete with restore capability.
  - Add Redis or in-memory caching for dashboard data.
  - Add a notification bell/dropdown in the header.
  - Add calendar view for tasks and shifts.
  - Add multi-language support (i18n).
  - Containerize with Docker for easy deployment.
  - Add CI/CD pipeline configuration.
